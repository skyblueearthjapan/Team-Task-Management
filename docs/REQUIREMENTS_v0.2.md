## 1. 概要

機械設計技術部のスタッフ約6名のタスクを一元管理する Web アプリケーション。Google スプレッドシートを DB として活用し、Google Apps Script (GAS) で WebApp として提供する。日々の予定（ガントチャート）・作業内容・前日までの作業報告を記録し、必要に応じてローカル PC の Outlook に**下書きを作成**または**直接送信**する仕組みを持つ。送信主体はリクエスト単位でスプレッドシート上から選択する。

参考: 姉妹プロジェクト [Electrical-Work-Scheduling](https://github.com/skyblueearthjapan/Electrical-Work-Scheduling) — **デザインとガント操作感のみ参照**。機能要件は本書を正とする。

## 2. 目的とスコープ

### 2.1 目的

- 機械設計技術部スタッフ全員のスケジュール・作業状況を可視化する
- 日次の作業報告を Outlook 経由でチーム共有しやすくする（**下書き経由の手動送信**を既定とし、必要時のみ自動送信）
- 工番／製品マスタを再入力させず、選択ベースで素早く記録できるようにする

### 2.2 スコープ

**含む**
- ガントチャートによる週次／月次スケジュール管理
- 日次の「本日の作業内容」「前日までの作業報告」入力
- 工番マスタ・スタッフマスタ・作業内容マスタとの連携
- スタッフ単位での「メール下書き作成 / 送信」依頼機能（Outlook 経由）
- ローカル常駐 Python EXE による Outlook 連携

**含まない**
- 工数集計・原価管理
- 顧客側への直接配信
- モバイル専用 UI（PC ブラウザ前提）

## 3. 利用者

| 区分 | 想定人数 | 操作範囲 |
|---|---|---|
| 機械設計技術部スタッフ | 約6名 | 自分の予定・作業内容・報告を入力／全員分を閲覧 |

```
├──────────────────────────────────────────────────────┤
│ 上部：ガントチャート                                   │
│   行 = スタッフ約6名                                   │
│   列 = 日付（週／2週／月の切替）                        │
│   セル = ドラッグで期間バーを作成・移動・伸縮            │
├──────────────────────────────────────────────────────┤
│ 下部：作業内容 / 前日までの作業報告                     │
│   スタッフ別タブ または 横並びカラム                    │
│   1人あたり最大 5〜6 行                                │
│   各行: 期間 / 工番 / 納入先 / 製品名 / 作業内容 / 詳細  │
│   末尾: [メール下書き作成 / 送信] ボタン                 │

```

### 5.1 上部：ガントチャート

- 行: 機械設計技術部スタッフ約6名（Staff マスタ）
- 列: 日付軸。表示期間 1週間 / 2週間 / 1ヶ月 切替（既定: 1週間）
- 操作: ドラッグでバー作成／両端で期間調整／本体ドラッグで日程移動／ダブルクリックで詳細編集
- 同一スタッフ内の重複は自動レーン分割
- 土日・祝日はグレーアウト
- 楽観的UI + 数秒デバウンス保存

### 5.2 下部：作業内容・報告欄

- 表示単位: 日付ごとに「何日から何日まで」の期間表示
- スタッフ別に最大 **5〜6項目**まで入力可能
- 各行の入力フィールド:

| 項目 | 入力方法 |
|---|---|
| 期間 | 日付ピッカー or ガントから自動連携 |
| 工番 | プルダウン（工番マスタ） — 一般作業時は「（一般作業）」 |
| 納入先 | 自動入力（工番マスタから引当） |
| 製品名 | 自動入力（工番マスタから引当） |
| 作業内容 | プルダウン（作業内容マスタ） |
| 詳細／自由記述 | テキスト |
| 区分 | 「本日の作業内容」 / 「前日までの作業報告」 |

- ガントで設定した期間は、対応する報告行の右側に**予定バー**として表示

### 7.4 メール送信機能（**v0.2 で全面更新**）

#### 7.4.1 UI フロー

1. スタッフ行末尾の **「メール下書き作成 / 送信」ボタン**を押下
2. §5.3 のモーダルが表示される
3. モーダル内で以下を確認・選択:
   - 作業スタッフ名と登録メールアドレスの**一致確認**（§7.4.2）
   - 送信モード: **下書き作成 / 直接送信**（既定: 下書き作成）
   - 送付先プレビュー
   - 本文差し込みデータのプレビュー
4. **「依頼を登録する」**で `MailQueue` に `status='pending'` で登録

#### 7.4.2 スタッフとメールアドレスの一致確認プロセス（**新設**）

「ボタンを押した瞬間」と「EXE が処理する直前」の二重チェックを行う。

| タイミング | チェック内容 | NG 時の挙動 |
|---|---|---|
| ① モーダル表示時（GAS 側） | `targetStaffId` の現在の Staff マスタの `email` を取得し、UI に明示 | メールアドレス未登録の場合は依頼ボタン非活性 |
| ② 依頼登録時（GAS 側） | `targetStaffName` と `targetStaffEmail` を**スナップショットとして MailQueue に保存** | スナップショット保存に失敗したら登録キャンセル |
| ③ EXE 取得直後 | `MailQueue.targetStaffEmail` と Staff マスタの最新 `email` を再照合 | 不一致なら `status='failed'`／`errorMessage='email_mismatch'`／管理者通知 |
| ④ 直接送信モード時 | EXE が **Outlook の現在のプロファイル送信者アドレス**と `targetStaffEmail` を照合（同一PCで複数スタッフが共有する場合の誤送信防止） | 不一致なら `status='failed'`／`errorMessage='outlook_profile_mismatch'` |

> ④ は「実行 PC とスタッフの紐付け」を運用上どうするかに依存（§12 で確認）。

## 8. 外部連携

### 8.1 ローカル送信エージェント (Python EXE)

- **役割**: Sheets を監視 → モードに応じて Outlook に下書き作成 / 直接送信
- **実装**:
  - `pywin32` の `win32com.client.Dispatch("Outlook.Application")` で COM 起動
  - 下書き: `mail.Save()` → 既定の「下書き」フォルダに保存
  - 送信: `mail.Send()` → 送信トレイ → Outlook が送信
- **動作環境**: 業務 PC（Outlook がインストール済み・ログイン済み・Windows）
- **認証**: Sheets API（Google Cloud のサービスアカウント or OAuth デスクトップフロー）— §12 で確認
- **常駐方法**: Windows タスクスケジューラ起動（ログオン時／PC 起動時）— §12 で確認
- **ログ**: ローカルファイル（`%LOCALAPPDATA%\TeamTaskMail\logs\YYYYMMDD.log`）と `MailLog` シートの両方
- **死活監視**: `Settings` シートに `lastHeartbeat` を1分ごとに書き込み、UI に「エージェント稼働中」のバッジ表示
- **ヘルプ動作**:
  - 同 ID を複数 EXE が掴まないよう `pickedBy` + 一意な「処理中」マークで楽観的ロック
  - 一定時間（例: 5分）`picked` のまま放置されたものは再 `pending` 化（タイムアウト復旧）

## 10. セキュリティ・運用上の注意（**v0.2 で追記**）

- **直接送信モードの誤爆防止**: モーダルの送信モード既定値は **「下書き作成」**。直接送信を選ぶ操作は明示的にラジオを切り替える必要あり。
- **メールアドレス不一致時の挙動**: GAS 側の Staff マスタが正本。EXE 側は照合のみ行う（書き換え不可）。
- **テンプレート改変**: EXE ハードコードのため、テンプレート変更時は再ビルド・再配布。配布管理（バージョン番号・更新方法）は §12 で確認。
- **複数スタッフの送信ボタン押下**: 同一 Outlook プロファイルから別スタッフの「直接送信」を行うと送信元が混乱する可能性あり → §7.4.2 ④ の照合で防止。
